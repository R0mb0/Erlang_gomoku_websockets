Gomoku Engine - Integration Tests

This document outlines the step-by-step testing procedure for the Erlang Backend. It demonstrates the initialization of the actors, state management, and the interaction between the Gomoku Game Master and the underlying Tuple Space Engine.

Phase 1 Bootstrapping the Architecture

1

Start the Application Environment

Navigate to the gomoku_backend directory and start the Rebar3 shell to automatically compile and load the modules.

cd .\gomoku_backend\
rebar3 shell
2

Boot the Tuple Space Engine

Initialize the tuple_space gen_server. This creates the ETS table in memory that will act as our distributed shared database.

tuple_space:start_link().
3

Boot the Gomoku Game Master

Initialize the gomoku_server. This actor manages the game rules, turns, and interfaces with the Tuple Space.

gomoku_server:start_link().

Phase 2 Testing Game Logic & State Validation

4

Initialize the Board

Demonstrates a massive asynchronous injection of tuples. The server generates 400 empty cells {cell, X, Y, empty} for the 20x20 grid.

gomoku_server:init_board().
5

Test State Protection (Wrong Turn)

Expected: Error

Context for Report: The server's initial state assigns the first turn to Player 1. If Player 2 attempts a move, the handle_call callback must reject it, proving that state is strictly maintained in memory. 

gomoku_server:make_move(player_2, 5, 5).

Output: {error,not_your_turn}

6

Test Valid Move & Tuple Injection

Expected: Success

Context for Report: Player 1 makes a valid move. The server accepts it, switches the active turn to Player 2, and forwards the move tuple to the Tuple Space via an asynchronous out. 

gomoku_server:make_move(player_1, 5, 5).

Output: {ok,move_accepted}

7

Verify Tuple Space Status

Uses a non-destructive read (rd) with wildcards to print the current state of the board, confirming the piece was successfully injected into the ETS table. 

gomoku_server:print_board().
